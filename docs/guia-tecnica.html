Este documento describe cómo se construyó el prototipo actual de MEDACCER, qué piezas son reales, qué es placeholder, y exactamente qué se necesita para llevarlo a producción: arquitectura, integraciones paso a paso, costos y cronograma.

00 — ResumenResumen ejecutivo

MEDACCER es un CRM y agente conversacional (WhatsApp) para consultorios médicos en Colombia. El producto combina tres capas:

🤖

Agente IA

Atiende WhatsApp 24/7, agenda citas, cobra, confirma, hace triage, recuerda controles.

📋

CRM médico

Panel para consultorio: pacientes, citas, conversaciones, historia clínica, facturación, RDA.

🔌

Integraciones

WhatsApp, Google Calendar, Wompi, MinSalud FHIR, n8n, ERP (Siigo/Alegra).

EstadoPrototipo completo, navegable y realista. Todo el frontend (landing + CRM + onboarding + checkout + blog) está funcional con datos mock. El siguiente paso es conectar el backend real siguiendo esta guía.

model Consultorio {
  id            String   @id @default(cuid())
  nombre        String
  nit           String?  @unique
  especialidad  String
  planId        String   // essential | pro | enterprise
  createdAt     DateTime @default(now())
  users         User[]
  patients      Patient[]
  appointments  Appointment[]
  integrations  Integration[]
}

model User {
  id             String  @id @default(cuid())
  consultorioId  String
  email          String  @unique
  role           Role    // OWNER | DOCTOR | RECEPTIONIST | ADMIN
  clerkId        String? @unique
  consultorio    Consultorio @relation(fields: [consultorioId], references: [id])
}

model Patient {
  id            String   @id @default(cuid())
  consultorioId String
  nombre        String
  documento     String   // cédula
  documentoTipo String   // CC | TI | CE | PA
  telefono      String
  email         String?
  nacimiento    DateTime?
  eps           String?
  consentLey1581 Boolean @default(false)
  consentDate   DateTime?
  metadata      Json?    // alergias, medicamentos, flags
  appointments  Appointment[]
  conversations Conversation[]
  records       ClinicalRecord[]

  @@index([consultorioId, documento])
  @@index([consultorioId, telefono])
}

  id            String   @id @default(cuid())
  consultorioId String
  patientId     String
  doctorId      String
  inicio        DateTime
  fin           DateTime
  estado        AppointmentStatus  // SCHEDULED | CONFIRMED | CANCELLED | NO_SHOW | COMPLETED

  notas         String?
  servicioId    String
}

model Conversation {
  id            String   @id @default(cuid())
  consultorioId String

  status        String   // active | resolved | escalated

  conversationId String

  toolCalls      Json?

model ClinicalRecord {
  id            String   @id @default(cuid())
  consultorioId String

  appointmentId String?

  transcripcion String?  // Whisper

  procedimientos Json?   // CUPS

model Integration {
  id            String  @id @default(cuid())
  consultorioId String

RLS Activa Row Level Security en Postgres para que ningún query pueda leer datos de otro consultorio, incluso si hay un bug en la API. Política: USING (consultorio_id = current_setting('app.consultorio_id')::uuid).

03 — MigraciónDel prototipo a producción

El código actual vive en index.html + archivos .jsx cargados por Babel en el browser. Esto sirve para prototipar pero no escala. Pasos para migrar:

1

Crear el monorepo con Turborepo

Estructura definida arriba. Permite compartir tipos TS entre front y back, deploys atómicos y un solo pnpm install.

~2 días · 1 senior
2

Migrar .jsx → .tsx con Vite

Copiar cada archivo del prototipo a apps/web/src/pages/. Añadir tipos. Reemplazar hash router por React Router. Los componentes visuales (landing, phone mockup, CRM) se reutilizan 95%.

~5 días · 1 frontend senior
3

Setup NestJS + Prisma + Postgres

Scaffold con nest new, añadir Prisma, definir schema del punto anterior, correr migraciones. Montar Auth con Clerk.

~3 días · 1 backend senior
4

Conectar CRM a API real

Reemplazar los useState con datos mock por TanStack Query calls. Empezar por /patients, /appointments, /conversations. Cada pantalla del CRM ya tiene su estructura de datos clara en el prototipo — solo hay que reemplazar el origen.

~2 semanas · 1 full-stack
5

Montar workers con BullMQ

Un worker por integración. Cada uno consume una queue de Redis. Separados para escalar independientemente.

~3 días · 1 backend
6

Deploy a AWS con Terraform

Infra as code desde el día uno. Tres ambientes: dev (costo ~$80/mes), staging (~$200/mes), production.

~1 semana · 1 devops

El agente de IA (la pieza central)

Esta es la parte más crítica del producto. En el prototipo es mock. Así se implementa real:

Arquitectura del agente

// packages/ai-sdk/src/agent.ts
import OpenAI from 'openai';
import Anthropic from '@anthropic-ai/sdk';

export class MedaccerAgent {
  constructor(private consultorioId: string) {}

  async handleMessage(msg: IncomingMessage): Promise<AgentResponse> {
    // 1. Cargar contexto del paciente + consultorio
    const ctx = await this.loadContext(msg.from);

    // 2. RAG: buscar servicios, precios, políticas relevantes
    const rag = await this.retrieveKnowledge(msg.text, ctx);

    // 3. Construir system prompt con guardrails
    const system = buildSystemPrompt({
      consultorio: ctx.consultorio,
      especialidad: ctx.consultorio.especialidad,
      patient: ctx.patient,
      knowledge: rag,
      tools: AVAILABLE_TOOLS,
    });

    // 4. Llamar LLM con function calling
    const response = await this.llm.complete({
      model: 'gpt-4o-mini',
      system,
      messages: ctx.history,
      tools: [
        checkAvailability,
        bookAppointment,
        cancelAppointment,
        sendPaymentLink,
        escalateToHuman,
        searchPatient,
      ],
    });

    // 5. Ejecutar tool calls (si los hay)
    if (response.toolCalls?.length) {
      for (const call of response.toolCalls) {
        await this.executeTool(call, ctx);
      }
    }

    // 6. Guardar en BD + responder por WhatsApp
    await this.persistMessage(response);
    return response;
  }
}

Tools que debe tener el agente

checkAvailability

Consulta Google Calendar + reglas del consultorio. Devuelve slots disponibles.

bookAppointment

Crea cita en BD, evento en Calendar, envía confirmación al paciente.

sendPaymentLink

Genera link Wompi, lo envía, escucha webhook de confirmación.

escalateToHuman

Marca conversación como escalada. Notifica recepcionista. Pausa el agente.

searchPatient

Por cédula, teléfono o nombre. Devuelve ficha con consentimiento Ley 1581 validado.

requestMedicalRecord

Adjunta últimos N encuentros relevantes para que el agente tenga contexto clínico.

Guardrails obligatorios

  • No dar diagnósticos: el system prompt debe prohibirlo explícitamente. Escala a humano si el paciente insiste.
  • No recetar medicamentos: nunca sugerir dosis, marcas o sustitutos.
  • Validar consentimiento Ley 1581 antes de almacenar o procesar cualquier dato personal.
  • Detección de urgencias: palabras clave ("dolor de pecho", "sangrado", "suicidio") → respuesta inmediata con número de emergencia + escalamiento.
  • Validación de outputs: cada tool call pasa por Zod schema antes de ejecutarse.
  • Auditoría: cada interacción guardada con timestamp, modelo usado, tokens consumidos, costo.

04 — IntegracionesIntegraciones paso a paso

WhatsApp Business Cloud API 3-5 días

Meta provee una API oficial (gratis hasta 1.000 conversaciones/mes, luego ~$0.05 USD por conversación iniciada por el negocio en Colombia). Las iniciadas por el usuario en las últimas 24h son gratis.

Pasos de implementación
  1. Crear Meta Business Account en business.facebook.com. Verificar el negocio con NIT, cámara de comercio, dominio medaccer.com.
  2. Crear WhatsApp Business Account (WABA) dentro del Business Manager.
  3. Agregar número. Cada consultorio puede traer el suyo (Bring Your Own Number) o comprar uno via Twilio.
  4. Aprobar "Display Name": el nombre que verá el paciente. Meta lo revisa en 1-2 días.
  5. Embedded Signup Flow: integra un flujo en tu app donde el consultorio autoriza con su cuenta de Facebook y le llega el permiso a MEDACCER. Ver docs oficiales.
  6. Registrar webhook: endpoint https://api.medaccer.com/webhooks/whatsapp/:consultorioId. Meta envía un challenge GET — responder con hub.challenge.
  7. Aprobar plantillas de mensajes (para iniciar conversaciones fuera de ventana 24h): "Recordatorio de cita", "Confirmación", "Link de pago". Revisión 24h.
  8. Verificar firma HMAC-SHA256 en cada payload entrante:
import crypto from 'crypto';

function verifyMetaSignature(rawBody: Buffer, signature: string, appSecret: string) {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', appSecret)
    .update(rawBody)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(expected)
  );
}

Enviar mensajes: POST a graph.facebook.com/v19.0/{PHONE_NUMBER_ID}/messages con el access token del consultorio.

GotchaRate limits: Meta limita a 80 mensajes/segundo por número en Tier 1. Si creces, subes de tier automáticamente. Diseña la queue con backoff exponencial.

Google Calendar API 2 días

Pasos de implementación
  1. Crear proyecto en Google Cloud Console.
  2. Habilitar Google Calendar API.
  3. Crear OAuth 2.0 Client ID (Web application). Redirect URI: https://app.medaccer.com/oauth/google/callback.
  4. Solicitar verificación del app (para scopes sensitive) — Google tarda 1-3 semanas. Mientras, el app funciona con hasta 100 test users.
  5. Scopes: https://www.googleapis.com/auth/calendar.events.
  6. Guardar refresh token cifrado. Access token expira cada hora — refresh automático.
  7. Usar Calendar Push Notifications (webhooks) para detectar cambios que hace el doctor manualmente en su Calendar.

Wompi (pagos Colombia) 5-7 días

Pasos de implementación
  1. Registro en comercios.wompi.co. Requiere cuenta Bancolombia empresarial.
  2. Sandbox keys inmediatas. Probar todo el flujo antes de producción.
  3. Documentación legal: RUT, cámara de comercio, certificación bancaria, documento del representante legal. Aprobación ~5 días hábiles.
  4. Integración: Widget (más rápido) o API directa (más control). Recomendado: API directa para que el paciente no salga del flujo.
  5. Webhook a /webhooks/wompi. Validar firma con event secret.
  6. Estados posibles: APPROVED | DECLINED | VOIDED | ERROR | PENDING. Polling opcional si el webhook tarda.
  7. Comisión: 2.65% + $900 COP por transacción. Factible cargar al consultorio o descontar del producto.

Alternativas: Mercado Pago (más fácil pero menos local), ePayco, PayU.

MinSalud RDA — HL7 FHIR 4-6 semanas

La integración más compleja. Resolución 1888 obliga reporte de RDA (Registro Diario de Atenciones). Detalle:

  1. Registrar el prestador en la plataforma de interoperabilidad del MinSalud.
  2. Obtener certificado digital (Certicámara o Andes SCD). ~$200 USD/año por cada prestador. Puede ser individual o corporativo.
  3. Implementar FHIR R4: recursos Patient, Encounter, Condition (CIE-10), Observation, Procedure (CUPS), Practitioner, Organization.

05 — ComplianceSeguridad y legal

Ley 1581 de 2012 (datos personales Colombia)

MEDACCER maneja datos sensibles de salud. La Ley 1581 y su decreto reglamentario (1377/2013) son obligatorios.

Obligaciones concretas

  • Registrar la base de datos personales en la SIC (Superintendencia de Industria y Comercio) via RNBD. Obligatorio para empresas con activos >100.000 UVT.
  • Política de tratamiento de datos publicada y aceptada por el titular antes de recopilar. El prototipo ya lo muestra en signup y en el primer mensaje del bot.
  • Consentimiento expreso para datos sensibles (salud). No vale aceptación tácita.
  • Finalidad específica: no se puede usar el dato para algo distinto a lo informado.
  • Derecho de habeas data: el titular puede consultar, actualizar, rectificar, suprimir sus datos. Responder en <10 días hábiles.
  • Oficial de Protección de Datos: persona designada. Puede ser interno o contratar servicio externo.
  • Reporte de incidentes: si hay una fuga, notificar a la SIC y a los titulares afectados en <15 días.

09 — DevOpsCI/CD y observabilidad

Sin CI/CD automatizado, los deploys serán manuales, lentos y propensos a romper cosas. Con observabilidad, sabrás qué falla antes de que un cliente te escriba enojado.

Pipeline de CI/CD recomendado (GitHub Actions)

Cada push a main dispara: lint → tests → build → deploy a staging. Cada tag v* dispara deploy a producción con aprobación manual. Tiempo total: ~6-8 minutos.

on:

10 — TestingTesting y QA

Un consultorio médico no perdona bugs. Si el bot agenda mal una cita o si la RDA falla, pierdes clientes y arriesgas multas. Estrategia de testing en pirámide:

Casos de prueba críticos (no negociables)

  1. Bot agenda cita en slot disponible → Google Calendar evento creado, paciente recibe confirmación
  2. Bot rechaza slot ocupado → ofrece 2 alternativas reales (no inventadas)
  3. Pago Wompi exitoso → factura electrónica generada, paciente notificado
  4. Pago Wompi fallido → cita NO se confirma, paciente recibe nuevo link en 5min
  5. RDA push exitoso → estado en BD = aceptado, ID de MinSalud guardado
  6. RDA push rechazado → reintentar 3x con backoff, alertar al sysadmin si falla
  7. Mensaje de paciente con tono de alarma → escala a doctor en < 30s
  8. Multi-tenant aislamiento → consultorio A nunca ve pacientes de consultorio B (incluye SQL injection)
  9. Habeas data delete → borrado completo en < 7 días, audit log conservado
  10. Rate limiting → 100 msgs/min por consultorio, 429 al exceder

11 — OperaciónRunbooks operacionales

Cuando algo falle a las 3am, no quieres reinventar el procedimiento. Estos son los 6 incidentes más probables y cómo resolverlos paso a paso.

RB-01 — Webhook WhatsApp no responde

Síntoma: alertas de Better Uptime · pacientes no reciben respuesta · queue acumulando.
Diagnóstico (5 min):
  1. railway logs --service api | grep webhook — buscar errores recientes
  2. curl https://api.medaccer.com/health — verificar API arriba
  3. Meta Business Manager → WhatsApp → Configuration → Webhook → "Test"
  4. Verificar META_VERIFY_TOKEN en env vars (no rotó accidentalmente)
Solución:
  1. Si API caído: railway up --detach
  2. Si verify token rotó: re-suscribir webhook desde Meta Business Manager
  3. Si Meta side: revisar metastatus.com
Postmortem template: docs/postmortems/YYYY-MM-DD-webhook-down.md

RB-02 — RDA push a MinSalud fallando masivamente

Síntoma: dashboard de RDA muestra > 20% rechazados en 1h.
Diagnóstico:
  1. Verificar fecha expiración cert: openssl pkcs12 -in cert.p12 -nokeys | openssl x509 -noout -dates
  2. Validar bundle JSON contra validator.fhir.org
  3. Revisar status MinSalud: prestadores.minsalud.gov.co
Solución:
  1. Si certificado expiró: renovar en Certicámara (1 día). Mientras, pausar push y notificar consultorios.
  2. Si bundle inválido: revisar último cambio del prompt IA, posible regresión en estructura
  3. Si MinSalud caído: encolar en DLQ, reintentar cada 30min

RB-03 — Costo OpenAI disparado

Síntoma: alerta de Datadog: gasto/hora > $20.
Diagnóstico: probable bucle infinito o un consultorio enviando spam.
  1. Query: SELECT consultorio_id, SUM(tokens) FROM ai_calls WHERE created_at > NOW() - INTERVAL '1 hour' GROUP BY 1 ORDER BY 2 DESC LIMIT 5
  2. Identificar consultorio outlier
Solución:
  1. Activar rate limit temporal: UPDATE consultorios SET ai_max_tokens_hour = 5000 WHERE id = 'xxx'
  2. Contactar al cliente, investigar (¿hijo del doctor jugando con el bot? ¿bot de spam atacando?)
  3. Si es ataque: bloquear número en Meta + agregar a denylist

RB-04 — Base de datos lenta (queries > 1s)

Síntoma: p95 latencia API disparada.
Diagnóstico:
  1. SELECT * FROM pg_stat_activity WHERE state = 'active' ORDER BY query_start
  2. Buscar queries lentas en pg_stat_statements
  3. Verificar índices: EXPLAIN ANALYZE de las queries top 5
Solución:
  1. Kill queries colgadas: SELECT pg_terminate_backend(pid)
  2. Agregar índices faltantes (post-mortem en docs)
  3. Si DB saturada: scale up Railway plan (1 click) o agregar read replica

RB-05 — Paciente reporta brecha de privacidad

Síntoma: usuario dice "vi mensajes de otro paciente".
Acción inmediata (P0, todo el equipo):
  1. Pausar el consultorio afectado: UPDATE consultorios SET status = 'paused' WHERE id = 'xxx'
  2. Capturar logs y screenshots — evidencia legal
  3. Identificar root cause: query sin WHERE consultorio_id, bug de cache, error de Clerk session
  4. Notificar SIC en < 15 días si confirmamos brecha (Ley 1581 art. 17)
  5. Contactar al paciente afectado en < 24h con explicación + qué se hizo
  6. Patch deployment + audit completo

RB-06 — Pérdida total de DB primaria

Síntoma: Railway DB inaccesible > 5 min.
Plan B:
  1. RTO objetivo: 1 hora · RPO objetivo: 5 min (snapshot continuo)
  2. Promote read replica → primary (railway db promote <replica-id>)
  3. Update DATABASE_URL en env vars
  4. Redeploy: railway up
  5. Verificar: hacer un mensaje de prueba
Backups: snapshots automáticos cada 24h en S3 (retención 30 días) + WAL streaming continuo a réplica.

13 — IAPrompts del agente IA

El agente conversacional usa un patrón de system prompt + tools + memoria. El system prompt cambia según la especialidad del consultorio. Aquí los prompts reales (versión simplificada):

Sistema base (común a todas las especialidades)

Eres {nombre_asistente}, asistente virtual del consultorio {nombre_consultorio}.
Trabajas para {nombre_profesional}, {titulo_profesional} en {especialidad}.

REGLAS INVIOLABLES:
1. NUNCA das diagnósticos. Si te lo piden, responde: "Eso lo puede evaluar mejor el/la {titulo} en consulta."
2. NUNCA recomiendas tratamientos farmacológicos no prescritos por {nombre_profesional}.
3. Si detectas señales de URGENCIA MÉDICA, escalas inmediatamente y le dices al paciente: "Esto necesita atención inmediata. Por favor vaya a urgencias o llámenos al {telefono_urgencias}."
4. NUNCA inventas precios, horarios, o disponibilidad. Solo usas la información de las tools que tienes.
5. NUNCA respondes nada que no esté en tu base de conocimiento. Prefieres "déjeme verificarlo" antes que inventar.

TONO:
- Profesional pero cálido. Ni robot, ni demasiado familiar.
- Vouseo en Colombia (usar "usted" por defecto). Tutear solo si el paciente tutea primero.
- Frases cortas. Bullets cuando hay opciones.
- Emojis suaves: 📅 🩺 ✅ 💊 — máximo 1-2 por mensaje.

Recepción de mensajes (POST) — con verificación HMAC

// 2. Mensajes entrantes (debe responder < 20s o Meta reintenta)
waRouter.post('/webhook', async (req, res) => {
  // Validar firma HMAC SHA256 (X-Hub-Signature-256)
  const signature = req.headers['x-hub-signature-256'] as string;
  const expected = 'sha256=' + crypto
    .createHmac('sha256', process.env.META_APP_SECRET!)
    .update(JSON.stringify(req.body))
    .digest('hex');

  if (!crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))) {
    logger.warn({ headers: req.headers }, 'WA invalid signature');
    return res.sendStatus(401);
  }

  // Respuesta inmediata (Meta penaliza si tardamos > 20s)
  res.sendStatus(200);

  // Procesar async via queue (BullMQ + Redis)
  try {
    const entries = req.body.entry || [];
    for (const entry of entries) {
      for (const change of entry.changes || []) {
        const messages = change.value?.messages || [];
        const phoneNumberId = change.value?.metadata?.phone_number_id;

        for (const msg of messages) {
          await messageQueue.add('incoming', {
            phoneNumberId,
            from: msg.from,
            messageId: msg.id,
            type: msg.type,  // text, image, audio, document, location, button
            content: extractContent(msg),
            timestamp: parseInt(msg.timestamp) * 1000,
            raw: msg,
          }, {
            attempts: 3,
            backoff: { type: 'exponential', delay: 2000 },
            removeOnComplete: { age: 3600, count: 1000 },
            removeOnFail: { age: 86400 },  // 24h en DLQ
          });
        }
      }
    }
  } catch (err) {
    logger.error({ err, body: req.body }, 'WA webhook processing error');
    // No re-throw: ya respondimos 200, error tracking via Sentry
  }
});

function extractContent(msg: any): string {
  switch (msg.type) {
    case 'text': return msg.text.body;
    case 'image': return `[IMAGE:${msg.image.id}]`;
    case 'audio': return `[AUDIO:${msg.audio.id}]`;
    case 'document': return `[DOC:${msg.document.id}]`;
    case 'button': return msg.button.text;
    case 'interactive': return msg.interactive.button_reply?.title || msg.interactive.list_reply?.title;
    default: return '[UNSUPPORTED]';
  }
}

    wa_message_id: messageId,

Antes de empezar, necesito tu autorización para tratar tus datos personales según la Ley 1581 de Colombia. ¿Aceptas?` },

Validación previa al envío

Antes de mandar a MinSalud, validar contra el validador oficial:

// apps/api/src/rda/validator.ts
import axios from 'axios';

export async function validateBundle(bundle: any): Promise<ValidationResult> {
  const response = await axios.post(
    'https://validator.fhir.org/validate',
    bundle,
    {
      headers: {
        'Content-Type': 'application/fhir+json',
        'Accept': 'application/fhir+json',
      },
      params: { profile: 'https://terminology.minsalud.gov.co/StructureDefinition/RDA-CO' },
      timeout: 10000,
    }
  );
  const issues = response.data.issue || [];
  return {
    valid: issues.every((i: any) => i.severity !== 'error' && i.severity !== 'fatal'),
    errors: issues.filter((i: any) => i.severity === 'error'),
    warnings: issues.filter((i: any) => i.severity === 'warning'),
  };
}

Push firmado a MinSalud

import https from 'https';
import fs from 'fs';
import axios from 'axios';

const agent = new https.Agent({
  pfx: fs.readFileSync(process.env.MINSALUD_CERT_PATH!),
  passphrase: process.env.MINSALUD_CERT_PASSWORD,
  rejectUnauthorized: true,
});

export async function pushRDA(bundle: any): Promise<PushResult> {
  // 1. Validar antes de mandar
  const validation = await validateBundle(bundle);
  if (!validation.valid) {
    throw new RDAValidationError(validation.errors);
  }

  // 2. Push con cert mTLS
  const response = await axios.post(
    'https://prestadores.minsalud.gov.co/api/rda/v1/Bundle',
    bundle,
    {
      httpsAgent: agent,
      headers: {
        'Content-Type': 'application/fhir+json',
        'Authorization': `Bearer ${await getMinsaludToken()}`,
      },
      timeout: 30000,
    }
  );

  return {
    estado: 'aceptado',
    minsalud_id: response.data.id,
    fecha_push: new Date(),
  };
}

WITH conversaciones_7d AS (
  SELECT id, escalada
  FROM conversaciones
  WHERE created_at > NOW() - INTERVAL '7 days'
)
SELECT
  COUNT(*) AS total,
  COUNT(*) FILTER (WHERE NOT escalada) AS resueltas_bot,

-- No-show rate por consultorio
SELECT
  c.nombre,
  COUNT(ci.*) AS total_citas,
  COUNT(*) FILTER (WHERE ci.estado = 'no_show') AS no_shows,
  ROUND(100.0 * COUNT(*) FILTER (WHERE ci.estado = 'no_show') / NULLIF(COUNT(ci.*), 0), 2) AS rate_pct
FROM consultorios c
LEFT JOIN citas ci ON ci.consultorio_id = c.id
  AND ci.fecha > NOW() - INTERVAL '30 days'
GROUP BY c.id, c.nombre
ORDER BY rate_pct DESC;

18 — GlosarioGlosario técnico y de negocio

Términos técnicos

BullMQ
Sistema de colas Node.js sobre Redis. Para procesar mensajes WhatsApp en background sin bloquear el webhook.
FHIR R4
HL7 Fast Healthcare Interoperability Resources versión 4. Estándar JSON para intercambio de datos clínicos. MinSalud lo exige.
HMAC SHA-256
Hash con clave secreta para verificar que un webhook viene realmente de Meta (no de un atacante).
JWT
JSON Web Token. Token de autenticación firmado, contiene claims como consultorio_id y user_id.
mTLS
Mutual TLS. Autenticación mutua con certificados — MinSalud requiere mTLS para enviar RDAs.
Multi-tenant
Una sola instancia del software sirve a múltiples clientes (consultorios), aislados por consultorio_id en cada query.
p50 / p95 / p99
Percentiles de latencia. p95=400ms significa "el 95% de las requests se resuelven en ≤ 400ms".
RTO / RPO
Recovery Time Objective (tiempo máx para volver) / Recovery Point Objective (datos máx perdidos). Para MEDACCER: RTO 1h, RPO 5min.
Webhook
URL que recibe eventos en tiempo real (ej: WhatsApp llama a /api/webhooks/whatsapp cuando llega un mensaje).

Términos de negocio

ARR
Annual Recurring Revenue. Ingresos anuales recurrentes (suscripciones).
CAC
Customer Acquisition Cost. Cuánto cuesta conseguir un cliente nuevo (marketing + ventas / # nuevos clientes).
Churn
% de clientes que cancelan en un periodo. Mortal para SaaS — más del 5% mensual y la empresa no escala.
LTV
Lifetime Value. Cuánto dinero genera un cliente promedio en toda su vida con MEDACCER.
MRR
Monthly Recurring Revenue. Ingresos mensuales recurrentes.
NRR
Net Revenue Retention. % de revenue que se conserva año a año, considerando upsells y churn. > 100% = los clientes existentes pagan más con el tiempo.
No-show
Paciente que agenda pero no asiste. Costo brutal en consultorios — slots desperdiciados.

Términos clínico-legales (Colombia)

EPS
Entidad Promotora de Salud. Las "aseguradoras" del sistema de salud colombiano.
Habeas data
Derecho del paciente a saber qué datos tienes de él, corregirlos o eliminarlos. Plazo legal: 7 días.
Ley 1581 de 2012
Ley colombiana de protección de datos personales. Equivalente al GDPR. Multas hasta 2.000 SMMLV.
RDA
Resumen Digital de Atención. Documento clínico que MinSalud exige por cada atención (Resolución 1888/2025).
RIPS
Registros Individuales de Prestación de Servicios. Reportes mensuales que las IPS hacen al MinSalud.
RM
Registro Médico. Número de registro profesional ante el MinSalud, equivalente a la matrícula profesional.
SIC
Superintendencia de Industria y Comercio. Quien sanciona violaciones a Ley 1581.

git init
git add .
git commit -m "Initial commit — MEDACCER prototype"

# Crear repo privado en github.com/tu-usuario/medaccer
git remote add origin git@github.com:tu-usuario/medaccer.git
git branch -M main
git push -u origin main

Deploy rápido del prototipo a Vercel

  1. Ir a vercel.com/new
  2. Import del repo GitHub recién creado
  3. Framework: "Other" (es HTML estático)
  4. Build command: vacío. Output directory: .
  5. Deploy. URL pública en ~30s.
  6. Comprar dominio medaccer.com (Namecheap/Google Domains ~$12/año) y apuntar a Vercel.
