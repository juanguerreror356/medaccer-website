Guía operativa de conexión.

El stack mínimo para operar MediFlow con costo real de $8 USD/mes al inicio — qué cuenta crear, en qué orden, qué clave generar, dónde pegarla, y cómo verificar que funciona. Sin sobreingeniería, sin dependencias innecesarias.

Stack: Next.js · Supabase · Gemini · WhatsApp Cloud · n8n Costo MVP: $8 USD/mes Tiempo setup: 6-10 horas total Persona requerida: 1 (vos)

01Pricing en COP vs USD: la decisión

Investigación del mercado SaaS colombiano (Alegra, Siigo, Loggro, ContaPyme, Cuenti, +20 herramientas analizadas): el 100% de los SaaS que apuntan a PYMES y profesionales en Colombia muestran sus precios en COP, no en USD. Esto no es opinión — es lo que el mercado eligió.

Lo que dice la evidencia

Siigo$145.993 / $191.327 / $207.869 COP/mesPYMES contables

Veredicto para MediFlow

Pricing público en COP. Costos internos en USD. Tu cliente es un doctor colombiano que cobra consultas en pesos, paga arriendo en pesos, y mentalmente compara contra Alegra ($59.900) o un asistente humano ($1.800.000). Mostrar "$59 USD" agrega fricción de cálculo (¿4.000? ¿4.200? ¿el dólar mañana?) y proyecta "extranjero, caro". Mostrar $249.000 COP/mes es el lenguaje nativo.

Reglas concretas a aplicar en el landing y la guía

  • Landing público: 100% COP. Sin USD a la vista en pricing, ahorros, ni ROI. Ejemplo: "Recuperás $1.200.000 COP/mes en consultas que se cancelaban".
  • Métricas de mercado/inversión: COP con paréntesis USD si el número es muy grande (Ej: "Mercado total: $2.4 billones COP ($600M USD)").
  • Costos de infra (interno): USD, porque las facturas reales llegan en USD (Vercel, Supabase, Gemini cobran tarjeta en dólares). Vos lo convertís a COP para tu contabilidad.
  • Dashboard admin tuyo: dual — MRR en COP (lo que cobrás) y costos en USD (lo que pagás), con margen calculado en %.
  • Facturación a clientes: COP exclusivamente. Pasás por Wompi (que liquida en COP a tu cuenta de banco colombiana).

02Stack revisado: lo que entra y lo que sobra

El stack que tenías era un stack de Series A ($700-1.400 USD/mes) montado encima de un MVP de 1 persona. El stack revisado es un stack de bootstrapping centralizado en Supabase: una sola plataforma hace auth, DB, storage, realtime y edge functions.

$0/mes hasta 100 GB bandwidth

✅ Supabase (Free → Pro)

~$1-10 USD/mes

06Supabase: el corazón del backend

Supabase es el reemplazo de 4 servicios en uno: Postgres + Auth + Storage + Realtime. Si entendés bien Supabase, el resto del stack es plomería. Esta es la sección más larga porque es la más importante.

Crear el proyecto

  1. supabase.com → Sign up con GitHub.
  2. New Project. Nombre: mediflow-prod. Database password: generá uno fuerte y guardalo en un password manager (no lo vas a poder recuperar). Región: South America (São Paulo) — la más cercana a Colombia, latencia ~80ms desde Bogotá vs ~140ms desde us-east-1.
  3. Pricing tier: Free. Tenés 500MB DB, 1GB storage, 50.000 monthly active users de Auth. Suficiente hasta cliente 30-40.
  4. Esperá 2 minutos a que se aprovisione. Vas a ver Project URL y anon/service_role keys en Settings → API.

Las 3 claves que necesitás

VariableDónde se usaSensibilidad
NEXT_PUBLIC_SUPABASE_URLFrontend + backendPública (OK exponer)
NEXT_PUBLIC_SUPABASE_ANON_KEYFrontend (browser)Pública (RLS la protege)
SUPABASE_SERVICE_ROLE_KEYBackend (server-side only)Secret crítico — bypass de RLS. Nunca al browser.

Row Level Security (RLS) — crítico

Sin RLS, cualquier doctor puede leer pacientes de cualquier otra clínica si conoce un UUID. Con RLS, la base de datos rechaza la query a nivel SQL. Es el secreto del multi-tenant seguro. 

-- Activar RLS en todas las tablas
ALTER TABLE tenants ENABLE ROW LEVEL SECURITY;
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE pacientes ENABLE ROW LEVEL SECURITY;
ALTER TABLE citas ENABLE ROW LEVEL SECURITY;
ALTER TABLE mensajes ENABLE ROW LEVEL SECURITY;

-- Helper: tenant del usuario actual
CREATE OR REPLACE FUNCTION current_tenant_id() RETURNS UUID AS $$
  SELECT tenant_id FROM users WHERE id = auth.uid() LIMIT 1;
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Helper: ¿es super_admin? (vos como CEO)
CREATE OR REPLACE FUNCTION is_super_admin() RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM users
    WHERE id = auth.uid() AND rol = 'super_admin'
  );
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Policy: cada tenant ve solo lo suyo, super_admin ve todo
CREATE POLICY "tenant_isolation" ON pacientes
  FOR ALL USING (
    is_super_admin() OR tenant_id = current_tenant_id()
  );

-- Repetir el patrón para citas, mensajes
CREATE POLICY "tenant_isolation" ON citas
  FOR ALL USING (is_super_admin() OR tenant_id = current_tenant_id());

CREATE POLICY "tenant_isolation" ON mensajes
  FOR ALL USING (is_super_admin() OR tenant_id = current_tenant_id());

Storage para audios y fotos

-- Crear bucket "media" desde Supabase Dashboard:
-- Storage → New bucket → name: media → Public: NO

-- Policy: solo el tenant dueño puede leer/escribir su carpeta
CREATE POLICY "tenant_media_access" ON storage.objects
  FOR ALL USING (
    bucket_id = 'media'
    AND (storage.foldername(name))[1] = current_tenant_id()::text
  );

Las rutas serán media/{tenant_id}/audios/2026-04-28-abc.ogg. La policy garantiza que el doctor de la clínica A no puede listar archivos de la B.

Crear lib/supabase/server.ts (para Server Components y API routes):

import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export const createClient = async () => {
  const cookieStore = await cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll: () => cookieStore.getAll(),
        setAll: (toSet) => toSet.forEach(({ name, value, options }) =>
          cookieStore.set(name, value, options))
      }
    }
  );
};

Página de login con magic link (Supabase Auth)

Crear app/login/page.tsx:

'use client';
import { useState } from 'react';
import { createClient } from '@/lib/supabase/client';

export default function Login() {
  const [email, setEmail] = useState('');
  const [sent, setSent] = useState(false);
  const supabase = createClient();

  const handleLogin = async (e) => {
    e.preventDefault();
    await supabase.auth.signInWithOtp({
      email,
      options: { emailRedirectTo: `${window.location.origin}/auth/callback` }
    });
    setSent(true);
  };

  return sent
    ? <p>Revisá tu correo. Te mandamos el link de acceso.</p>
    : (
      <form onSubmit={handleLogin}>
        <input type="email" value={email} onChange={e => setEmail(e.target.value)} />
        <button>Enviar link</button>
      </form>
    );
}

El magic link es la mejor UX para doctores: no tienen que recordar contraseña. Click en el correo y entran. Supabase maneja el SMTP gratis con su dominio (después podés conectar Resend o SendGrid para mejor deliverability).

09WhatsApp Cloud API (Meta directo)

Esta es la integración más delicada porque depende de Meta y tiene 4 pasos burocráticos. Plan: empezar el día 1 — Meta tarda 24-72h en aprobar.

Setup

  1. Crear Meta Business Account en business.facebook.com. Te pide nombre legal + email + país (Colombia).
  2. Crear App en Meta for Developers: developers.facebook.com → My Apps → Create App → "Business" type. Nombre: "MediFlow Bot".
  3. Agregar producto WhatsApp: dentro de la app, "Add product" → WhatsApp → Setup. Te asigna un número de prueba gratis (formato +1 555…) que sirve para desarrollo.
  4. Generar System User Token permanente: Business Settings → System Users → Add → "Admin". Generar Access Token con permisos whatsapp_business_messaging y whatsapp_business_management. Sin expiración. Guardar en password manager.
  5. Migrar tu número real (cuando arranques con clínicas pagas): WhatsApp Manager → Add phone number → seleccionar tu eSIM dedicada. Meta envía un código por SMS. Verificás. Pasa por proceso de "Display Name" review (24-48h).
  6. Configurar webhook: Configuration → Webhooks → URL: https://mediflow.co/api/whatsapp/webhook. Verify token: una string random tuya (ej. mfw_2026_xK9pL2). Subscribe a: messages y message_status.

Endpoint en Next.js

Crear app/api/whatsapp/webhook/route.ts:

import { NextRequest, NextResponse } from 'next/server';

// Verificación inicial (Meta llama una vez con GET)
export async function GET(req: NextRequest) {
  const params = req.nextUrl.searchParams;
  const mode = params.get('hub.mode');
  const token = params.get('hub.verify_token');
  const challenge = params.get('hub.challenge');

  if (mode === 'subscribe' && token === process.env.WHATSAPP_VERIFY_TOKEN) {
    return new NextResponse(challenge, { status: 200 });
  }
  return new NextResponse('Forbidden', { status: 403 });
}

// Recepción de mensajes (POST)
export async function POST(req: NextRequest) {
  const body = await req.json();

  // Reenviar al webhook de n8n para procesarlo
  await fetch(process.env.N8N_WEBHOOK_URL!, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body)
  });

  // Meta requiere 200 en <5s o reintenta
  return new NextResponse('OK', { status: 200 });
}

Función agente conversacional

Crear lib/agent/gemini.ts:

import { GoogleGenerativeAI } from '@google/generative-ai';

const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY!);

const SYSTEM_PROMPT = `Eres MediFlow, asistente del consultorio de [DOCTOR_NAME].

REGLAS:
- Solo agendas, confirmas o cancelas citas. NO das diagnósticos.
- Si el paciente describe síntomas graves (dolor de pecho, sangrado, mareo severo), respondes:
  "Eso requiere atención médica directa. ¿Llamo al doctor o prefieres ir a urgencias?"
- Hablas en tono cercano colombiano, sin tutear excesivamente.
- Confirmas datos antes de agendar: nombre, cédula, fecha, hora.
- Si la franja no está disponible, ofreces 2 alternativas.
- Cierres con: "Confirmado, te mando recordatorio 24h antes."

CONTEXTO DEL PACIENTE:
- Nombre: {{nombre}}
- Última cita: {{ultima_cita}}
- Próxima cita: {{proxima_cita}}
- Tratamiento activo: {{tratamiento}}
`;

export async function chatWithPatient(historial, mensajeNuevo, contextoPaciente) {
  const model = genAI.getGenerativeModel({
    model: 'gemini-1.5-flash',
    systemInstruction: SYSTEM_PROMPT.replace('{{nombre}}', contextoPaciente.nombre)
      .replace('{{ultima_cita}}', contextoPaciente.ultimaCita ?? 'ninguna')
      .replace('{{proxima_cita}}', contextoPaciente.proximaCita ?? 'ninguna')
  });

  const chat = model.startChat({ history: historial });
  const result = await chat.sendMessage(mensajeNuevo);
  return result.response.text();
}
Tool calling (lo poderoso)

Gemini soporta function calling nativo. Le definís funciones (buscar_disponibilidad, crear_cita, cancelar_cita) y el modelo decide cuándo llamarlas. El paciente dice "agéndame el martes 4pm" → Gemini llama buscar_disponibilidad("martes 4pm") → tu código consulta Google Calendar → si libre, crear_cita. Esto evita parsing manual y reduce errores.

11Groq Whisper (transcripción de audios)

Pacientes mandan notas de voz contando síntomas. Whisper de OpenAI cuesta $6 USD/1000 min. Groq corre el mismo modelo a velocidad absurda y tiene free tier de 14.400 audios/día.

  1. console.groq.com → Sign up con Google → API Keys → Create.
  2. Pegar en Vercel: GROQ_API_KEY=gsk_...
  3. Instalar: 
    npm install groq-sdk
  4. Función: 
    import Groq from 'groq-sdk';

const groq = new Groq({ apiKey: process.env.GROQ_API_KEY });

export async function transcribirAudio(audioBuffer: Buffer) {
  const file = new File([audioBuffer], 'audio.ogg', { type: 'audio/ogg' });
  const { text } = await groq.audio.transcriptions.create({
    file,
    model: 'whisper-large-v3-turbo',
    language: 'es'
  });
  return text;
}

El audio de WhatsApp llega como URL en el webhook. Lo descargás con fetch, lo pasás a Groq, y le pasás el texto resultante a Gemini como mensaje normal.

12n8n self-hosted en Railway

n8n es el reemplazo de "escribir 30 routes en Next.js". Es Zapier open source, visual, con 400+ nodos. Self-hosted en Railway: $5 USD/mes. Hace toda la orquestación: WhatsApp → transcribir → Gemini → calendar → DB.

Por qué n8n y no escribir todo en Next.js

  • Visualizás el flujo. Cuando estés con el doctor mostrándole "así funciona", abrís n8n y ve la cosa correr en vivo. Imposible con código.
  • Logs por nodo. Si Gemini falló, ves el input/output exacto del nodo. Sin printf debugging.
  • Re-ejecutás un flujo fallido. El paciente no recibió respuesta porque Calendar tuvo timeout — re-corrés el flujo desde el nodo que falló, sin perder estado.
  • Plantillas. Cuando agregues "consultorio dental" como vertical, clonás el flujo, ajustás 3 prompts, listo.

Deploy en Railway

  1. railway.app → Sign up con GitHub → New Project → Deploy a Template → buscar "n8n".
  2. Railway provisiona n8n + PostgreSQL en 90s. Te da URL https://n8n-production-xxxx.up.railway.app.
  3. Conectar dominio propio: Settings → Networking → Custom Domain → n8n.mediflow.co. Crear registro CNAME en tu DNS.
  4. Primera vez: andá a n8n.mediflow.co. Te pide crear usuario admin. Email + password fuerte. Login.
  5. Activar autenticación básica HTTP (extra capa): Variables de entorno en Railway: 
    N8N_BASIC_AUTH_ACTIVE=true
N8N_BASIC_AUTH_USER=admin
N8N_BASIC_AUTH_PASSWORD=<password fuerte>
N8N_PROTOCOL=https
N8N_HOST=n8n.mediflow.co
WEBHOOK_URL=https://n8n.mediflow.co

El flujo principal: "WhatsApp Inbound"

Dentro de n8n, New Workflow → "WhatsApp Inbound". Nodos en orden:

[1] Webhook trigger → recibe POST de Next.js POST /webhook/wa-incoming expects: { entry: [{ changes: [...] }] } [2] Function (extract message) → saca el texto/audio msg = $json.entry[0].changes[0].value.messages[0] return { from: msg.from, type: msg.type, content: msg.text?.body, mediaId: msg.audio?.id } [3] IF node → ¿es audio? type === 'audio' ├── true → [4a] HTTP Request (Groq) → texto └── false → continúa con texto crudo [5] Postgres node (Supabase) → obtener/crear paciente INSERT INTO pacientes (whatsapp, tenant_id) VALUES (...) ON CONFLICT (whatsapp, tenant_id) DO NOTHING RETURNING * [6] Postgres node → últimos 10 mensajes (contexto) SELECT * FROM mensajes WHERE paciente_id = $1 ORDER BY created_at DESC LIMIT 10 [7] HTTP Request (Gemini) → generar respuesta POST https://generativelanguage.googleapis.com/... body: { contents: [historial + mensaje nuevo] } [8] IF node → ¿Gemini llamó tool "crear_cita"? └── sí → [9a] Google Calendar (insert event) [9b] Postgres (insert into citas) [10] HTTP Request (WhatsApp send) → responder al paciente POST https://graph.facebook.com/v18.0/.../messages [11] Postgres (log mensaje out)

Tiempo de implementar este flujo en n8n: ~3 horas la primera vez. Versus en código: ~3 días.

Conectar n8n con Supabase

Credentials → New → Postgres. Con la connection string que sale de Supabase → Settings → Database → Connection string → URI mode. Pegar.

Conectar n8n con Google Calendar

Credentials → New → Google Calendar OAuth2. Cada doctor que se onboardea hace click en "Conectar mi calendario" — eso lanza el flujo OAuth y guarda el token de ese doctor. n8n soporta múltiples credenciales por servicio.

17Flujo end-to-end real (paciente → cita confirmada)

Esto es lo que pasa cuando una paciente nueva escribe "hola, quiero pedir cita":

[T+0s] Paciente envía WhatsApp: "Hola, quiero pedir cita para el martes" [T+0.1s] Meta WhatsApp Cloud envía POST a https://mediflow.co/api/whatsapp/webhook payload: { entry: [{ changes: [{ value: { messages: [{...}] } }] }] } [T+0.2s] Next.js webhook retorna 200 OK inmediato (Meta requiere <5s) y forwardea el payload a https://n8n.mediflow.co/webhook/wa-incoming [T+0.3s] n8n: Function node extrae { from: '573001234567', content: 'Hola, quiero pedir cita...' } [T+0.4s] n8n: Postgres (Supabase) → buscar tenant por 'whatsapp_phone_number_id' que recibió el mensaje → buscar/crear paciente con whatsapp=573001234567 → si paciente nuevo: marcar consent_habeas_data=false (el bot lo pedirá) [T+0.5s] n8n: Postgres → SELECT últimos 10 mensajes del paciente para contexto [T+1.5s] n8n: HTTP Gemini → manda system prompt + historial + mensaje nuevo → Gemini responde: { tool_call: "buscar_disponibilidad", args: { dia: "martes" } } [T+2.0s] n8n: Google Calendar → freebusy() del calendario del doctor para el martes → respuesta: [10:00, 10:30, 14:00, 15:00] libres [T+2.5s] n8n: HTTP Gemini (segundo turn) → le mando el resultado de la tool → Gemini responde: "¡Hola! Sin humano. [T+15s] Paciente: "10am perfecto" [T+16s] Bot: "Perfecto. ¿Me confirmas tu nombre completo y cédula?" [T+45s] Paciente: "Camila Rodríguez, 1.234.567.890" [T+46s] Bot: "Listo Camila. 20Errores comunes (y cómo resolverlos)

Error / SíntomaCausaSolución
RLS policy violation al hacer SELECT El usuario logueado no tiene tenant_id en la tabla users. Asegurate de que en el flujo de signup/onboarding, después de auth.signUp, hagas un INSERT en users con el tenant correspondiente.
WhatsApp webhook no recibe POSTs Meta llamó GET para verificar y devolviste algo distinto al challenge. Verificar que tu route /webhook retorne exactamente params.get('hub.challenge') en GET. Mirar Vercel logs.
Mensajes WhatsApp tardan >10s en responder Cold start de Vercel + Gemini lento + Calendar lento (suma). Mover lógica pesada a n8n (que no tiene cold start). El webhook de Next solo hace forward + 200 OK.
Gemini responde en inglés System prompt no es lo suficientemente fuerte. Agregar al inicio: "SIEMPRE respondes en español colombiano. Nunca en otro idioma." y temperature: 0.3.
n8n se cae al ejecutarse muchos flujos Railway plan default tiene 512MB RAM. Subir a $10/mes (1GB) cuando pases 30 mensajes/min. Antes de eso, no es necesario.
Wompi: signature mismatch Cálculo del HMAC de integridad mal hecho. String a firmar: {reference}{amount-in-cents}{currency}{integrity-key}. SHA-256 hex. Sin espacios.
Vercel build falla con TypeScript errors Errores en código que next dev ignoraba. Correr npm run build localmente antes de pushear. Vercel es estricto. Si urge: typescript.ignoreBuildErrors: true en next.config (deuda técnica).
Audio de WhatsApp llega pero Groq dice "file empty" El audio de WA viene como media_id, no como URL directa. Hay que descargarlo primero con tu access_token. 2 pasos: (1) GET https://graph.facebook.com/v18.0/{media_id} retorna URL. (2) GET esa URL con Authorization: Bearer {token} retorna el binario.
