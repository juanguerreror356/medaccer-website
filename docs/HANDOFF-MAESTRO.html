05Lo difícil (mes 2-3)

1. Conectar webhook WhatsApp

Es la parte que más se traba. Tenés codigo-base/app/api/webhooks/whatsapp/route.ts con el código base. El flujo:

  1. Tu webhook está en Vercel: https://medaccer.co/api/webhooks/whatsapp.
  2. Meta necesita verificar el webhook con un token GET. El código ya lo tiene.
  3. Variables de entorno en Vercel: WHATSAPP_VERIFY_TOKEN (lo invento yo, ej "medaccer_verify_2026"), WHATSAPP_APP_SECRET, WHATSAPP_TOKEN, PHONE_NUMBER_ID.
  4. En Meta Business → WhatsApp → Webhooks → Configurar URL → meter la URL de Vercel + el verify token.
  5. Suscribirse a eventos: messages.
  6. Mandás mensaje desde tu celular al número de pruebas → Meta hace POST → Vercel recibe → guardás en Supabase.

2. Conectar Gemini con prompts médicos

Los prompts están en codigo-base/lib/ai/system-prompts.ts (uno por especialidad: dental, dermatología, psicología, estética, medicina general). Falta:

  1. Instalar SDK Gemini en el proyecto Next.js: npm i @google/generative-ai.
  2. En el webhook de WhatsApp, después de guardar el mensaje: llamar Gemini con el prompt de la especialidad del doctor + el historial reciente del paciente.
  3. Gemini responde — texto plano o con tool_calls (definidos en codigo-base/lib/ai/tools.ts: book_appointment, send_quote, handoff_to_human).
  4. Si hay tool_call: ejecutarla (insertar cita en Supabase, enviar handoff por email, etc).
  5. Mandar la respuesta de vuelta al paciente vía API de WhatsApp (POST a /messages).

3. Wompi en producción + suscripciones

  1. Cuando Wompi te apruebe el comercio, generás llaves de producción.
  2. En el frontend: Wompi Widget embebido en la página de checkout.
  3. El doctor mete tarjeta → Wompi tokeniza → guarda token en tu DB.
  4. Para suscripciones recurrentes: Wompi tiene Pagos por suscripción. Configurás un cron (Vercel Cron) que cobra el día 1 de cada mes.
  5. Webhook de Wompi: /api/webhooks/wompi recibe eventos de pago aprobado/rechazado.
  6. Si pago rechazado 3 veces seguidas: suspender bot del doctor automáticamente.

La Cloud API es gratis hasta 1000 conversaciones/mes. Después cobran por mensaje plantilla (~30-50 COP cada uno). Los mensajes que el paciente inicia son gratis durante 24h después de su último mensaje.

  1. En Meta Business Manager → WhatsApp Accounts → "Get started".
  2. Aceptás términos (los específicos de WhatsApp Business).
  3. Configurás Display Name: "MEDACCER" (lo van a ver los pacientes en el chat).
  4. Subís foto de perfil (logo cuadrado 640x640).
  5. Categoría: Health o Medical Services.
  6. Te dan número de pruebas + IDs.

Fase 4 · Tu número real

  1. Necesitás un número que no esté en WhatsApp normal. Si tu número personal lo está: tenés que dar de baja WhatsApp en ese número primero (perdés historial).
  2. Recomendación: compras un número virtual de Twilio (~$1 USD/mes), o usás un chip nuevo de Movistar/Claro (~$15.000 COP único).
  3. En Meta: agregás el número → te llega código por SMS → lo confirmás.
  4. El número queda bloqueado para WhatsApp normal — solo se usa por la API.
  5. Ese número es el que verán los pacientes. Si querés cambiarlo después, es papelera.

Fase 5 · Plantillas (templates)

Para iniciar conversación vos al paciente (no responder, iniciar), Meta exige plantillas pre-aprobadas. Ej: "Hola {{1}}, te recordamos tu cita el {{2}} a las {{3}}". Las creás en Meta Business → WhatsApp → Templates. Aprobación: 1-24h.

💡 Plantillas que vas a usar
  1. Recordatorio de cita (24h antes)
  2. Confirmación de cita (al agendarse)
  3. Cobro pendiente (después de la cita)
  4. Recuperación de presupuesto (3-7 días después de cotizar sin agendar)
  5. Encuesta post-consulta (12h después)

08Supabase · tu base de datos

Setup inicial

  1. supabase.com → New project.
  2. Nombre: medaccer-prod. Región: São Paulo. DB password: generala fuerte y guardala.
  3. Plan Free al inicio. Subís a Pro ($25 USD/mes) cuando tengas 50+ doctores.
  4. En "Project Settings → API": copiás URL, anon key (público, va al frontend), service_role key (PRIVADO, solo backend).
  5. En Vercel → Project → Settings → Environment Variables: agregás las 3.

Correr migraciones

  1. Dashboard de Supabase → SQL Editor → New query.
  2. Abrís codigo-base/migrations/001_init.sql en local.
  3. Copiás-pegás todo en SQL Editor → "Run".
  4. Si sale error: leelo, corregís, volvés a correr (los CREATE TABLE IF NOT EXISTS hacen que sea idempotente).
  5. Repetís con 002, 003, 004, 005 en ORDEN.
  6. Verificás en Table Editor que aparezcan 20+ tablas.

Row Level Security (RLS)

🔒 Esto es SÚPER importante

Sin RLS, cualquier doctor puede ver datos de otros doctores. Las migraciones ya tienen ENABLE ROW LEVEL SECURITY + policies. Verificá en cada tabla que esté el ícono del candado verde. Si falta, no la actives en producción.

Storage (para fotos de pacientes)

  1. En Supabase → Storage → New bucket.
  2. Nombre: patient-files. Public: NO (privado).
  3. Policies: solo el doctor dueño puede leer/escribir archivos donde el path empiece con su doctor_id.

13SEO · cómo aparecer en Google

Tu landing actual tiene SEO pobre porque es una SPA con todo el contenido en JSX (Google ve poco). Te paso 8 cambios concretos que disparan ranking.

1. Metadatos correctos (impacto: ALTO, esfuerzo: BAJO)

<title>MEDACCER · WhatsApp con IA para consultorios médicos en Colombia</title>
<meta name="description" content="Reduce no-shows del 25% al 8%. Tu bot agenda, cobra y recuerda citas en WhatsApp. 7 días gratis. Cumple Ley 1581.">
<meta property="og:image" content="https://medaccer.co/og.png">
<link rel="canonical" href="https://medaccer.co/">
<meta name="geo.region" content="CO">
<meta name="geo.placename" content="Bogotá">

2. Schema.org JSON-LD (impacto: ALTO, esfuerzo: BAJO)

Google muestra estrellas, precios, reviews directo en search. Agregás:

</script>

3. Server-Side Rendering (impacto: ALTO, esfuerzo: MEDIO)

Tu landing actual es JSX en cliente. Google indexa pero no idealmente. Cuando migrás a Next.js, Server Components renderizan HTML puro → Google ve todo. Sin tocar nada más, +30% indexación.

4. Sitemap + robots.txt (impacto: MEDIO, esfuerzo: BAJO)

# public/robots.txt
User-agent: *
Allow: /
Sitemap: https://medaccer.co/sitemap.xml

# app/sitemap.ts (Next.js auto-genera)
export default function sitemap() {
  return [
    { url: 'https://medaccer.co', lastModified: new Date() },
    { url: 'https://medaccer.co/precios', lastModified: new Date() },
    { url: 'https://medaccer.co/blog', lastModified: new Date() },
  ];
}

5. Blog SEO (impacto: ALTO, esfuerzo: ALTO)

Es el motor real de SEO. Pero tarda. Cada post bien hecho te trae 50-200 visitas/mes a 6 meses. Te recomiendo:

  • 15 posts en 3 meses, 1.500-2.500 palabras cada uno.
  • Keywords colombianas con menos competencia: "software citas dentista Colombia", "sistema agenda médica WhatsApp", "como reducir no-shows consultorio".
  • Título estructurado: H1 con keyword principal, H2 con sub-temas.
  • Mínimo 3 imágenes por post + alt text.
  • Link interno entre posts.

Cuando estés listo, te genero los 15 títulos + outlines basados en keyword research real.

6. Velocidad (impacto: ALTO, esfuerzo: BAJO)

  • Imágenes en formato WebP (no PNG/JPG). 80% menos peso.
  • Lazy loading: <img loading="lazy">.
  • Vercel Edge Network entrega tu sitio en <200ms desde Colombia.
  • Test: pagespeed.web.dev. Apuntá a >90 en mobile.

7. Backlinks (impacto: ALTO, esfuerzo: ALTO)

Que otros sitios médicos te enlacen. Estrategia mes 3-6:

  • Guest posts en blogs de doctores colombianos.
  • Aparecer en directorios: SiigoApps, Capterra LATAM, GetApp Colombia.
  • Notas de prensa cuando tengas 50 doctores.

8. Google Search Console (impacto: MEDIO, esfuerzo: BAJO)

  1. search.google.com/search-console.
  2. Agregás medaccer.co → verificás con TXT record en DNS.
  3. Subís el sitemap.
  4. Cada semana mirás "Performance": qué keywords te traen tráfico.

14Cómo me subís fotos

Acá no hay magia: hay 3 formas de pasarme imágenes. Te explico cada una con cuándo usarla.

Opción 1 · Pegar en el chat (más fácil)

En el chat, hacés drag & drop de la imagen al textarea, o copiás-pegás (Ctrl/Cmd + V). Yo la veo de inmediato y puedo analizarla. Ideal para: screenshots, fotos sueltas que solo querés que vea.

Opción 2 · Subir al proyecto y referenciar

Si querés que la imagen quede como asset del sitio (logo, foto de equipo, screenshot del producto que aparezca en la landing):

  1. En el chat, drag & drop de la imagen — pero pedís: "guardá esta imagen como assets/logo.png".
  2. Yo la guardo en la carpeta del proyecto.
  3. Después la referencio en el HTML: <img src="assets/logo.png">.

Opción 3 · Mediante GitHub (cuando ya esté el proyecto vivo)

Cuando el proyecto esté en GitHub:

  1. Subís la imagen al repo en una carpeta public/images/.
  2. Me das el link de raw GitHub o me decís el nombre del archivo.
  3. Lo uso desde el código.

17Cómo no quemar tokens

Vas a abrir un chat nuevo. Para que dure el doble:

  1. No subas todo el zip de una. Subí solo lo que necesites para esa tarea.
  2. Empezá con el prompt de contexto de la sección 01 — me ahorra ~20.000 tokens de explicación.
  3. Pedí cosas concretas, no abiertas. "Hacé X" mejor que "qué te parece esto".
  4. Cuando termines una tarea, cerralá explícitamente. Decí "perfecto, listo, vamos al siguiente". Eso hace que olvide los detalles intermedios y libere contexto.
  5. No pidas que recuerde más allá del proyecto activo. Cada tarea grande, chat nuevo.
  6. Si el chat se siente lento o repetitivo → es señal de que se está llenando el contexto. Cortá y abrí uno nuevo con un nuevo handoff.
✓ Resumen de un buen flujo de trabajo

1 chat = 1 milestone grande. Ej: "configurar Supabase + correr migraciones + verificar conexión". Cuando ese milestone está done, cierro chat, abro uno nuevo con handoff actualizado, sigo con el próximo milestone.
